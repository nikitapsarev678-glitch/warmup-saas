# Фаза 21 — Quality & Launch Checklist (качество «максимально правильно»)

> Это фаза‑чеклист. Делается после того, как MVP‑фазы реализованы.  
> Цель: продукт должен ощущаться надёжным, объяснимым и безопасным.

## 1) UX качество (ощущение продукта)
- [ ] Везде есть empty‑states (аккаунты/прокси/задачи/рассылки/лиды).
- [ ] Любая пауза/ошибка имеет `reason` + “что сделать” + CTA (продолжить/пополнить/проверить/сменить прокси).
- [ ] Прогресс виден: task/broadcast показывают этап, проценты, ETA (приблизительно), последние события.
- [ ] Landing: работает без регистрации, fast LCP, scrollytelling без лагов на mobile (fallback).

## 2) Надёжность очередей и runner
- [ ] `task_queue` дренится идемпотентно (retry safe).
- [ ] Любой runner action имеет таймаут и пишет error + stack summary в D1.
- [ ] Порционирование: большие рассылки/парсинг идут чанками, не упираются в GitHub Actions timeout.

## 3) Токены и деньги
- [ ] Списание токенов идемпотентно (`token_usage_events.idempotency_key` UNIQUE).
- [ ] При `balance=0` задачи реально не продолжают выполнение (fail closed).
- [ ] В UI видно: текущий баланс, что “съело” токены (последние 20 usage events).

## 4) Anti-ban и безопасность аккаунтов
- [ ] Прокси checker работает и подсвечивает dead/slow.
- [ ] SpamBot check не спамит и кеширует результат.
- [ ] Лимиты и паузы применяются на backend (Worker/runner), а не только в UI.

## 5) Безопасность (минимум)
- [ ] Никакие `session_string`, `api_hash`, `proxy.password` не попадают в ответы API.
- [ ] Все запросы multi-tenant (`WHERE user_id=?`) + авторизация на всех приватных endpoints.
- [ ] Rate-limit на опасные endpoints: import, parsing, start/stop, check-all.
- [ ] Webhook Robokassa строго проверяет подпись.

## 6) Наблюдаемость (минимум)
- [ ] Есть “системные” логи: последние 100 ошибок runner по user_id.
- [ ] Есть “продуктовые” логи: warmup_actions + broadcast_messages + parsing_jobs.

## 7) Канареечный запуск
- [ ] Фича-флаги на parsing/AI/группы‑каналы send (можно выключить без деплоя).
- [ ] 1–3 тестовых аккаунта для регресса (warmup + 10 DM + 1 follow-up).

## 8) Definition of Done (финальный)
- [ ] Все пункты из `SPEC.md` → “MVP Definition of Done” закрыты.
