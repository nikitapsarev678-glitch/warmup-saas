# Фаза 18 — Token Metering (списание по факту выполнения)

> Читай SPEC.md перед началом. Фазы 10, 16 должны быть выполнены (tokens, broadcasts).  
> Эта фаза формализует: за что списываем токены, как останавливаем задачи при нуле, как считаем usage.

## Цель
Токены списываются **по факту действий**, не “за создание задачи”.

---

## Модель списаний (предложение для MVP)

1) **AI‑операции** (дороже):
   - генерация шаблонов рассылки
   - генерация диалогов прогрева
2) **Обычные действия** (дешевле):
   - 1 отправленное DM
   - 1 диалог‑сообщение в прогреве
   - 1 инвайт/коммент/сторис (если появятся позже)

Нужно явно задать цену в токенах (константы в Worker) и хранить usage.

Рекомендуемая таблица цен (MVP, можно менять позже):
- `dm_sent` = 1 🪙
- `group_or_channel_send` = 2 🪙
- `warmup_action` = 1 🪙
- `ai_generate_template` = 20 🪙
- `ai_generate_dialog` = 5 🪙 / сообщение

---

## D1: usage ledger

Добавить миграцию `worker/migrations/0010_token_usage.sql`:
- `token_usage_events` (user_id, action, units, tokens_spent, ref_type/ref_id, created_at)

Правило: **идемпотентность** — одно действие не должно списаться дважды (например через уникальный `idempotency_key`).

---

## Остановка задач при нуле

Если `balance < required`:
- mark task/broadcast as `paused_reason = 'tokens'`
- записать событие
- (фаза 17) отправить уведомление

---

## Acceptance criteria

- [ ] Есть таблица `token_usage_events`
- [ ] Списание работает идемпотентно
- [ ] При нуле токенов задача паузится и не продолжает отправку
