# Фаза 17 — Автоматизация + Уведомления (Cron + Telegram)

> Читай SPEC.md перед началом. Фазы 1, 2, 10, 16 должны быть выполнены (database, auth, tokens, broadcasts).  
> Эта фаза добавляет: расписания (cron), batch-check аккаунтов, уведомления пользователю в Telegram (настраиваемо).

## Цель
Сделать “как Contez + enterprise UX”:
- cron‑задачи (проверки, догон, health)
- уведомления в Telegram по важным событиям

---

## Что уведомляем (MVP — включено по умолчанию)

1) **Токены закончились** → задачи ставятся на паузу + отправить уведомление
2) **Аккаунт ушёл в spam_block / banned** → уведомление + рекомендация действий
3) **Batch-check завершён** → итог (сколько активных/ошибок)

Пользователь должен уметь выключить конкретные типы уведомлений.

---

## D1: настройки уведомлений

Добавить миграцию `worker/migrations/0009_notifications.sql`:
- `user_notification_settings` (per user toggles)
- `notification_events` (лог отправленных событий, чтобы не спамить)

## Worker: cron

Добавить cron handlers (wrangler triggers) для:
- `batch_account_check` (например, раз в 6 часов)
- `followup_dispatch` (раз в 30 мин)
- `queue_poller` (опционально, если нужно чаще триггерить runner)

Также добавить ручную кнопку/эндпоинт `POST /accounts/check-all` (см. SPEC API) — запускает batch-check сразу.

---

## Telegram transport

Варианты:
1) **Telegram bot** (рекомендуется): отправка сообщений пользователю по `telegram_id` из `saas_users`.
2) Email (не нужен для MVP).

Анти-спам:
- дедупликация через `notification_events`
- лимит: не более N событий одного типа в час на пользователя
